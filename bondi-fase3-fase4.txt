2. Fase 3 — Producto robusto listo para clientes
El objetivo de la Fase 3 es convertir el sistema en un producto que una agencia puede mostrar en una demo y que un cliente puede usar de forma autónoma, sin necesidad de que la agencia esté presente. Las funcionalidades están ordenadas por impacto real en ventas y retención.
?? Objetivo de Fase 3
Un cliente de la agencia puede entrar al sistema con su propio usuario, ver su calendario, aprobar publicaciones, ver sus reportes y entender sus resultados — sin necesidad de WhatsApp ni de exportar nada manualmente.

2.3  Capa 3 — Robustez antes de escalar
Estos puntos no agregan funcionalidades visibles para el usuario final pero son obligatorios antes de ofrecer el sistema como producto de suscripción. Sin ellos, el sistema puede fallar o perder datos de clientes que pagan.
?  Multi-tenancy real
Hoy los datos viven en localStorage del browser. Si el usuario cambia de dispositivo, pierde todo. Se necesita backend con base de datos donde cada cliente tiene su espacio aislado accesible desde cualquier dispositivo.
?  Autenticación y roles
El sistema de login actual es básico. Para SaaS se necesitan al menos tres roles: Admin (agencia), Cliente (solo lectura de su cuenta) y Colaborador (puede editar pero no borrar).
?  Backup automático
Exportación periódica automática de todos los datos. Si alguien limpia el localStorage o hay un error, los meses de métricas e historial no se pierden.
?  Control de costos de IA
Contador de tokens consumidos por cliente y por mes. Límite configurable por plan. Alerta antes de alcanzar el límite. Sin esto, un cliente que genera mucho contenido puede consumir el presupuesto de IA de toda la agencia.


3. Fase 4 — Arquitectura SaaS de producción
La Fase 4 es el salto de "herramienta interna" a "producto SaaS real". No se trata de optimizar el código JavaScript existente — es mover la lógica crítica al servidor para soportar múltiples usuarios simultáneos, garantizar disponibilidad y habilitar el modelo de suscripción con cobro automático.
?? Punto de partida importante
El sistema actual es una SPA con localStorage. Eso significa que toda la lógica, los datos y las llamadas de IA corren en el browser del usuario. Funciona perfecto para una agencia con 5-10 clientes. Para escalar a 100+ usuarios concurrentes, la lógica crítica debe vivir en el servidor.
3.1  Stack tecnológico recomendado
CAPATECNOLOGÍARAZÓNBackend principalNode.js + FastifyMismo lenguaje que el sistema actual (JS). Fastify maneja 30.000-50.000 req/s en un servidor modesto. Ecosistema enorme. Para escalar a 500k+ usuarios, migrar a Go.Base de datos principalPostgreSQLTransaccional, soporte nativo para JSON, escala muy bien con índices correctos. Ideal para usuarios, clientes, contenido, métricas, planes y pagos.Caché y colasRedisResultados de IA ya generados (evitar regenerar), tokens de sesión, métricas calculadas. Reduce el costo de IA a la mitad. También maneja la cola de jobs de generación.FrontendReact o SvelteEl HTML/CSS/JS vanilla actual migra gradualmente. Svelte produce bundles muy livianos — ideal para carga rápida. React tiene más ecosistema si luego se necesita colaboración externa.Infraestructura inicialRailway / RenderDeploy automático con cada commit. Sin configurar servidores manualmente. Cuando se superan los 500 usuarios activos, la migración a AWS / GCP es natural.CDN y seguridadCloudflareAssets estáticos servidos desde la red de Cloudflare (no desde el servidor). Certificado SSL automático. DDoS protection. Rate limiting de red.
3.2  Arquitectura de alta disponibilidad
Con esta arquitectura, el sistema no cae cuando falla una instancia y puede absorber picos de tráfico sin degradar la experiencia del usuario.
Load balancer + múltiples instancias
En lugar de un solo servidor, corren 3 o más instancias del backend detrás de un load balancer (Nginx o Caddy). Si una instancia falla, el tráfico se redirige automáticamente a las otras. El usuario nunca ve un error de servidor caído.
Cola de generación de IA (BullMQ + Redis)
En lugar de que cada usuario espere bloqueado mientras la IA responde, las peticiones de generación van a una cola. Un worker procesa en background y cuando termina notifica al usuario vía WebSocket. Esto resuelve los tres problemas de deuda técnica: no bloquea el hilo principal, permite streaming de respuesta y la cola absorbe picos de tráfico.
Gateway de IA agnóstico de proveedor
Una capa de abstracción única que redirige a OpenAI, Claude o Gemini según disponibilidad y precio. Si un proveedor cae o sube sus precios, el switch es en un solo archivo. El resto del sistema no sabe con quién habla.
Capacidad estimada con esta arquitectura: 200-300 usuarios activos simultáneos en un servidor de $50/mes. Para 1.000 usuarios simultáneos: 3-4 instancias, costo de $150-200/mes de infraestructura — cubierto con 10-15 suscripciones del plan básico.

3.5  Seguridad de nivel producción
?  JWT con refresh tokens
Las sesiones no se guardan en el servidor — se emiten tokens firmados con expiración corta (15 minutos) y refresh tokens de larga duración. Si un token es robado, expira solo.
?  Encriptación de datos sensibles
Las API keys de OpenAI/Claude de cada cliente se guardan encriptadas en la base de datos, no en texto plano. Nunca en el frontend.
?  Audit log completo
Cada acción relevante queda registrada: quién hizo qué y cuándo. Útil para resolver disputas y para cumplimiento normativo.
?  Rate limiting por plan
Cada plan tiene límite de llamadas a la IA por mes. Redis lleva el contador en tiempo real. Si un usuario del plan básico llega al límite, el sistema le muestra el mensaje de upgrade antes de hacer la llamada — no después de gastar el token.
?  HTTPS obligatorio
Cloudflare maneja el certificado SSL automáticamente. Sin excepción, nunca HTTP.

3.6  Plan de migración incremental
La migración de Fase 3 a Fase 4 no tiene que ser de golpe. La estrategia más segura es en 4 etapas sin romper lo que ya funciona:
#ETAPAQUÉ SE HACE1Autenticación al backendLogin real con base de datos. El frontend sigue igual pero en lugar de guardar en localStorage, guarda en el servidor. Impacto en el usuario: ninguno visible.2Migración de datosEl calendario, las métricas y los clientes se mueven a PostgreSQL. El frontend hace fetch al backend en lugar de leer localStorage. Permite acceso multi-dispositivo.3IA al backend con colasLa generación de contenido se mueve al servidor con BullMQ. El frontend envía el job y escucha el resultado vía WebSocket. Elimina los timeouts y permite streaming.4Portal de gestiónAplicación separada que consume la misma API del backend. Dashboard de negocio, gestión de licencias y pagos con Stripe.
